# Room Escape — 2인 협동 방탈출

언리얼 엔진 5.5 / C++ 기반의 **2인 협동 방탈출 게임**입니다.
두 플레이어가 서로 다른 방(Room A / Room B)에서 시작해, 각자 얻은 단서를 채팅으로 공유하며 퍼즐을 풀고 함께 탈출해야 합니다.

- **개발 기간**: 2026.06.16 ~ 2026.07.21 (내일배움캠프 언리얼 8기 · CH4 팀 프로젝트)
- **장르**: 1인칭 협동 퍼즐 어드벤처
- **플랫폼**: Windows
- **멀티플레이**: 리슨 서버(Listen Server) 방식 2인 협동

## 팀원 및 역할

| 이름 | 담당 |
|---|---|
| 이길호 | 1인칭 캐릭터·GAS 상호작용 시스템, 유리미로/피아노 퍼즐, 게임 진행(Progression) 시스템, 이벤트 알림 시스템, 레벨 그레이박스 |
| 차지훈 | UI 프레임워크(CommonUI·WidgetManager), 메인메뉴/세션룸/HUD/팝업 UI, 채팅 시스템 |
| 김형인 | 폭탄 해체 퍼즐, 거울 방 퍼즐, Lock & Glow 퍼즐 및 리팩토링 |
| 한승재 | 세션 생성/검색/참가, 서버 목록 UI, 버튼·텍스트 스타일 등 UI 디자인 적용 |
| 김훈철 | 맵 제작 및 레벨 아트, 라이팅 |

## 게임 흐름

```
메인메뉴 → 세션 생성/참가 → 세션룸(역할 A/B 선택, Ready)
  → 게임맵 트래블 → [1부] 방 퍼즐 3종 클리어 → 지하 개방
  → [2부] 탈출 퍼즐(미로/타일) + 폭탄 해체
       ├─ 성공 → 탈출 엔딩
       └─ 실패 → 배드엔딩 연출 → 리스타트 포인트에서 2부만 재시작
```

## 주요 시스템 구조

### 공통 네트워킹 설계 원칙

모든 게임플레이 시스템이 아래 리슨 서버 패턴을 일관되게 따릅니다.

1. **판정은 전부 서버 권한** — `HasAuthority()` 검사 후에만 상태 변경. 오버랩·타이머 바인딩도 서버 한정
2. **상태는 복제 변수 + OnRep** — `ReplicatedUsing`으로 전파, OnRep에서 BP 연출 이벤트 호출
3. **리슨 서버 관용구** — 호스트에서는 OnRep이 호출되지 않으므로 서버 측 변경 함수가 OnRep을 직접 호출해 로컬 발화 보장
4. **일회성 연출은 Multicast RPC** — 함정 발동, 피아노 음 재생, 배드엔딩 시작 등 상태가 아닌 이벤트는 `NetMulticast`로 전파

### 캐릭터 · 상호작용 (GAS)

- `AREPlayerCharacter`: 1인칭 카메라/팔 메시 분리 렌더링, `UAbilitySystemComponent`(Mixed) 소유
- 상호작용: 입력 → `GA_Interact`(LocalOnly)가 **클라이언트 카메라 기준 라인트레이스** → `IREInteractable` 대상이면 `ServerInteract` RPC → 서버에서 거리 재검증 후 실행
- 손전등: `GA_Flashlight` → 서버 토글 → `bFlashlightOn` 복제. 빔 방향은 `GetBaseAimRotation` 기반으로 원격 플레이어도 동기화되며, 발광 도료 퍼즐이 이 빔 데이터를 활용

### 퍼즐 프레임워크

- `AREPuzzleManager`: `Locked → Active → Solved/Failed` 상태 머신(복제). 파생 매니저는 `HandlePuzzle*` virtual 훅만 오버라이드
- `AREPuzzleInteractableActor`: 매니저가 Active일 때만 조작 가능한 상호작용 요소 베이스 (프롬프트 위젯 포함)
- 퍼즐 요소(벽·레버·함정 등)는 레벨 배치 시 매니저를 지정하고 BeginPlay에서 자기 등록 → 매니저는 배치를 모르고 판정만 담당

### 퍼즐 목록

| 퍼즐 | 협동 구조 |
|---|---|
| 유리벽 미로 | 레버로 개폐되는 유리벽 미로. 함정 회피, 관찰자-이동자 협동 |
| 피아노 시퀀스 | A방이 멜로디를 듣고 B방이 셔플된 건반으로 입력. 정답은 서버만 보유 |
| 폭탄 해체 | 한 명은 매뉴얼, 한 명은 폭탄. 제한 시간 내 해체 실패 시 배드엔딩 |
| 거울 방 | 거울 단서 기반 입력 퍼즐 |
| Lock & Glow | 손전등 발광 도료 단서 + 다이얼 자물쇠 |
| 타일 경로 | 안내자-이동자 협동 타일 밟기 |

### 게임 진행 시스템 (Progression)

- `AREProgressionManager`가 방 퍼즐들의 `OnPuzzleSolved`를 구독해 스테이지(`RoomPhase → EscapePhase → Escaped`)를 전환
- 퍼즐·문·리스타트 포인트는 레벨 인스턴스의 Details 패널에서 배선(Wiring) — 코드 수정 없이 레벨 구성 가능
- 폭탄 실패 시: 배드엔딩 Multicast 연출 → 플레이어를 리스타트 포인트 A/B로 텔레포트 → **탈출 퍼즐만 리셋**(방 퍼즐 클리어 상태 유지)

### 세션 · UI · 채팅

- 세션: 세션 생성/검색/참가, 세션룸에서 역할(Alpha/Beta) 선택과 Ready. 선택한 역할은 GameInstance에 보존되어 게임맵 스폰 위치(`PlayerStartTag`)와 진행 시스템의 A/B 판정까지 관통
- UI: CommonUI 기반 레이어드 구조(`Primary / Gameplay / Overlay / Popup / VisualEffect`) + `LocalWidgetManager`(LocalPlayer 서브시스템)로 위젯 등록/조회 일원화
- 채팅: PlayerController의 `ChattingComponent`가 Server RPC → 전 플레이어 Client RPC로 중계. Enter 키로 입력 포커스 전환

### 이벤트 알림 시스템

- `URENotifySubsystem::NotifyEvent(태그, 메시지)` 단일 창구로 게임 이벤트를 GameplayTag 체계로 분류
- `[Server/Client]` 접두사가 붙는 `LogREEvent` 로그로 리슨 서버 디버깅 시 실행 주체를 즉시 구분

## 프로젝트 실행 안내

> **⚠️ 에셋 안내**: 대용량 아트 에셋(`Content/Resource`, 약 3.5GB)은 GitHub 용량 제한으로 저장소에 포함되어 있지 않으며 **팀 구글 드라이브로 별도 관리**됩니다. 저장소만 클론할 경우 일부 레벨의 아트 에셋이 누락될 수 있습니다.

1. 저장소 클론 후 구글 드라이브의 `Resource` 폴더를 `RoomEscape/Content/Resource`에 배치
2. `RoomEscape.uproject` 우클릭 → *Generate Visual Studio project files*
3. 솔루션 빌드(Development Editor) 후 에디터 실행
4. 멀티플레이 테스트: PIE → Net Mode `Play As Listen Server`, Players 2

## 회고

### 아쉬웠던 점

- **기획서 없이 출발한 초반 2주** — 기획서가 제대로 정리되지 않은 상태로 개발을 시작해, 프로젝트 초반 2주간 방향을 잡는 데 시간을 많이 소모했고 전체 진행 속도가 크게 느려졌다. 각자 만든 시스템이 하나의 게임으로 합쳐지는 시점도 그만큼 뒤로 밀렸다.
- **팀원 참여도 편차** — 중반 이후 팀원들의 개인 일정으로 참여도가 고르지 못했고, 일부 파트에 작업이 몰리면서 일정 관리와 협업 리듬 유지가 어려웠다.
- **구현하지 못한 기능들** — 방이 분리된 협동 게임의 핵심 재미가 될 수 있었던 **무전기(보이스/근접 통신) 기능**과, 엔딩·배드엔딩 연출을 위한 **시네마틱(레벨 시퀀스) 기능**을 일정상 시도하지 못한 것이 가장 아쉽다. 채팅으로 소통을 대체했지만 긴장감 있는 협동 경험 측면에서는 무전기가 있었다면 완성도가 달랐을 것이다.

### 배운 점

- **기획이 곧 속도다** — 기획서·공간 구조가 확정된 이후의 개발 속도는 초반과 비교가 되지 않았다. 다음 프로젝트에서는 그레이박스와 기획 문서를 먼저 확정하고 코드를 시작해야 한다는 것을 몸으로 배웠다.
- **일관된 네트워킹 패턴의 힘** — "서버 권한 판정 → 복제/OnRep → 호스트 직접 호출" 패턴을 초반에 정착시킨 덕분에, 퍼즐이 6종으로 늘어나도 네트워킹 버그가 특정 패턴 위반 지점으로 좁혀져 디버깅이 수월했다.
- **에셋/브랜치 관리의 중요성** — uasset 충돌, 커밋에 딸려 들어간 구버전 파일로 인한 코드 유실, 대용량 에셋의 저장소 관리 문제를 직접 겪으며 언리얼 팀 협업에서 에셋 관리 규칙(드라이브 분리, 브랜치 전략)이 코드만큼 중요함을 배웠다.
